# MedAsk 产品需求文档（PRD）

> 文档状态：面试展示版 / 当前实现基线
> 产品版本：规则版 MVP + 受控 LLM 辅助
> 最后更新：2026-07-19
> 说明：本文描述语言信息整理产品，不构成临床分诊、诊断或治疗方案。

## 1. 产品概述

MedAsk 是一个面向 18—65 岁成人的就医前症状信息整理工具。它通过最多 7 轮确定性追问，把用户零散的症状描述整理为可查看、修改、复制和打印的结构化摘要。

产品不回答“我得了什么病”，也不提供疾病概率、处方、药物、剂量、检查或治疗建议。明确风险表达由本地确定性规则在模型调用前处理；可选 LLM 只负责提出槽位候选，不能控制会话状态、风险结论或最终摘要。

### 1.1 一句话定位

帮助患者在就医前把症状信息说清楚，而不是让 AI 替代医生作出结论。

### 1.2 当前发布状态

| 能力 | GitHub Pages 规则版 | 开发 Mock | 真实 Provider |
|---|---|---|---|
| 发热 `fever` | 支持 | 支持 | 支持 |
| 咳嗽 `cough` | 支持 | 支持 | 支持 |
| 头痛 `headache` | 支持 | 支持 | 禁用 |
| 头晕 `dizziness` | 支持 | 支持 | 禁用 |
| 腹痛 `abdominal_pain` | 实验性 | 实验性 | 禁用 |
| 胸部不适 | 未开发 | 未开发 | 禁用 |

“支持”仅表示能够进行语言信息整理，不表示具备临床诊断或分诊能力。

## 2. 背景与问题定义

### 2.1 用户问题

用户通常知道“哪里不舒服”，但容易遗漏或混淆以下信息：

- 什么时候开始、目前是否仍存在；
- 当前数值与历史最高值；
- 伴随症状、否定症状与已经缓解的症状；
- 已经采取过的措施；
- 哪些信息不清楚、尚未获取或不适用。

### 2.2 数据问题

数据治理阶段发现医疗对话数据包含模板、重复、`title/ask` 错配、儿童或孕产妇问题，以及诊断、问药、检查解读和医院费用等非预问诊意图。医生 `answer` 还可能包含诊断和治疗内容，因此不能直接作为患者端回答依据。

产品由此选择：只使用患者侧信息提炼标签与边界；医生答案、原始患者数据和人工金标不进入患者端仓库。

### 2.3 产品机会

把“生成一个医学答案”改为“完成一次低负担、可回看、可交接的信息整理”，可以形成更小、更可验证且具备失败降级路径的 AI 产品问题。

## 3. 目标与非目标

### 3.1 产品目标

1. 用户能在最多 7 轮内形成一份忠实于原始输入的结构化摘要。
2. 明确风险表达在普通流程和 Provider 调用前被保守拦截。
3. 模型不可用、超时或输出非法时，用户仍能继续标准规则流程。
4. 用户能跳过、返回上一步、修改答案并安全清空会话。
5. 规则、模型候选和最终写入之间具有可解释、可测试的控制边界。

### 3.2 非目标

- 疾病诊断、鉴别诊断或疾病概率；
- 药物推荐、剂量、处方、检查或治疗建议；
- 替代医生或完整临床分诊；
- 处理儿童、孕期、产后或超过当前年龄范围的人群；
- 自动把 AI 标签升级为人工金标；
- 在没有审核知识库和内容责任机制时引入医学 RAG 问答。

## 4. 用户与核心场景

### 4.1 目标用户

- 年龄 18—65 岁；
- 希望在就医、在线联系人工服务或描述症状前整理信息；
- 主诉属于当前规则支持范围；
- 能理解产品是信息整理工具，而不是诊断系统。

### 4.2 核心场景

1. **普通信息整理**：用户选择主诉或输入一句话，按标准问题补齐信息并获得摘要。
2. **一次提供多项信息**：规则或受控 LLM 提取已有槽位，避免重复追问。
3. **否定、历史或已缓解**：系统区分“没有”“以前有”“已经好了”和当前肯定状态。
4. **风险前置**：命中明确风险表达后停止普通流程，提示人工或线下处理。
5. **失败降级**：Provider 不可用或候选未通过校验时继续标准问题。
6. **修改与回退**：用户修改前置答案后，条件槽位和摘要随之重算。

## 5. 用户旅程

```text
欢迎页
  ↓ 年龄与主诉范围检查
主诉识别
  ↓ 风险规则前置
最多 7 轮标准追问
  ↓ 可选模型候选 + Harness 二次校验
结构化摘要
  ↓
复制 / 打印 / 返回修改 / 返回首页
```

可能终态：

- `completed`：完成信息整理并生成摘要；
- `escalated`：命中保守风险表达，停止普通流程；
- `unsupported`：年龄、人群或主诉不在当前支持范围。

## 6. 功能需求

### 6.1 欢迎页与会话初始化

| ID | 需求 | 验收标准 |
|---|---|---|
| FR-01 | 展示产品定位与安全边界 | 普通用户可见“不提供诊断、用药或治疗建议” |
| FR-02 | 输入年龄与初始描述 | 年龄范围与 500 字上限同时在页面和控制器校验 |
| FR-03 | 提供快速主诉入口 | 仅展示当前规则版支持的主诉；腹痛明确标记实验性 |
| FR-04 | 处理不支持输入 | 不强行识别，进入 `unsupported` 并说明当前范围 |
| FR-05 | 静态模式无 API 依赖 | GitHub Pages 不请求 `/api/llm/*`，完整规则流程仍可运行 |

### 6.2 主诉识别与当前性

| ID | 需求 | 验收标准 |
|---|---|---|
| FR-06 | 识别当前肯定主诉 | 仅明确当前或近期本次症状进入普通流程 |
| FR-07 | 处理否定 | “没有发热”等不写成阳性主诉 |
| FR-08 | 处理历史与假设 | 纯历史或“如果以后……”不进入当前流程 |
| FR-09 | 处理近期已缓解 | 可进入本次信息整理，但摘要明确显示已缓解 |
| FR-10 | 处理局部转折 | “没有胸痛，但后来突然胸痛”按后半句当前风险判断 |
| FR-11 | 处理模糊表达 | 不能可靠映射时进入固定澄清，不猜测标签 |

### 6.3 槽位与 7 轮流程

1. 根据主诉加载槽位定义。
2. 已从首句获得的信息不重复询问。
3. `required` 和风险相关槽位优先。
4. `showWhen` 不成立的条件槽位标记为不适用，不计入缺失。
5. 主动跳过的信息进入“用户暂不清楚”。
6. 达到 7 轮仍未询问或未获取的信息进入“尚未获取”。
7. “已采取措施”属于交接摘要必问字段，必须在 7 轮内出现。
8. 多主诉按 `slotId` 共享起病时间、已采取措施等槽位，不重复询问。

### 6.4 编辑、导航与恢复

| ID | 需求 | 验收标准 |
|---|---|---|
| FR-12 | 返回上一步 | 恢复上一问题和原答案，不增加 `turnCount` |
| FR-13 | 修改已填信息 | 仅接受合法类型、范围和枚举值 |
| FR-14 | 重算条件槽位 | 修改前置答案后清理不再适用的子槽位答案 |
| FR-15 | 返回首页 | 必须确认清空；取消时保留会话 |
| FR-16 | 风险后不可回退 | `escalated` 后不能返回普通问诊绕过风险中断 |

### 6.5 风险前置

风险规则先于 Mock 或真实 Provider 执行。当前保守规则至少覆盖明确胸痛、明显呼吸困难、意识异常或昏厥，以及各已支持主诉中已经冻结的明确风险表达。

验收标准：

- 当前肯定风险表达命中后 `status=escalated`；
- 普通问诊停止，Provider 调用次数为 0；
- 否定、纯历史、已缓解、假设和无效模板不得触发升级；
- 页面只显示固定安全提示，不推断疾病原因。

### 6.6 结构化摘要

摘要只复述用户已经提供或回答的信息，并包含：

- 患者范围与核心主诉；
- 起病时间、当前状态、伴随信息；
- 已采取措施；
- 尚未获取；
- 用户暂不清楚；
- 是否触发风险升级；
- “仅用于预问诊信息整理，不是诊断结论”的声明。

底层枚举必须映射为中文显示，数值带正确单位。用户自述的既往诊断或开药信息必须使用“用户自述”前缀，不能改写成系统结论。

### 6.7 LLM 辅助

模型只作为候选提取器和非风险问题改写器：

1. 原始文本先通过风险规则。
2. 服务端根据自身规则重算允许槽位，客户端只能缩小范围。
3. 输出必须通过 Schema、槽位白名单、evidence 原文定位、局部上下文、置信度、类型、数值和已有答案冲突校验。
4. 只有当前肯定且通过所有校验的非风险候选，才可能由 Harness 写入。
5. 模型不能修改 `status`、`turnCount`、`maxTurns`、`required` 或 `showWhen`。
6. 失败、超时、限流和非法输出全部回退标准问题。

## 7. Function Calling 与 RAG 决策

### 7.1 Function Calling

项目实现并测试过唯一槽位提交工具 `submit_slot_extraction`。真实 DeepSeek 环境持续返回明确不支持错误，因此默认关闭 Strict Function Calling，使用 `json_object`，并保留完全相同的下游校验。

Function Calling 是输出约束方式，不代表语义正确，也不赋予模型业务控制权。

### 7.2 RAG

V1 不接入 RAG。当前核心问题是信息缺失与表达结构化，不是模型缺少医学知识；产品也不回答医学知识。

未来只有在以下条件同时满足时才考虑受控 RAG：

- 内容限于审核过的就诊准备或患者教育材料；
- 来源可追溯，有版本、过期和撤回机制；
- 无检索结果时安全降级，不让模型凭空补充；
- 检索内容不参与诊断、治疗、风险判断和状态机控制。

## 8. 非功能需求

### 8.1 安全与隐私

- API Key 只存在于服务端环境变量；
- 日志不记录用户原文、evidence、原始模型响应或完整 IP；
- 静态版不调用真实 Provider；
- 输入长度、Content-Type、CORS、限流和预算由服务端校验；
- 不复制原始医疗数据、人工金标或行级患者文本到产品仓库。

### 8.2 可用性

- 桌面和 390×844 手机视口可完成流程；
- 每轮展示当前进度、主诉标签和下一步操作；
- 网络失败不能造成白屏；
- 重复提交被阻止；
- 普通用户界面不显示 Provider、Schema、置信度、Trace 或内部规则 ID。

### 8.3 可测试与可解释

- 规则、槽位、风险和 Acceptance Policy 均有稳定 ID；
- 每次状态变化在会话内记录非敏感 Trace；
- 合成案例、未见集与真实 Provider 评测使用不同分母；
- 失败结果和禁用状态保留，不通过降低阈值换取发布。

## 9. 指标与发布门禁

### 9.1 当前基线

| 指标 | 当前结果 | 解释 |
|---|---:|---|
| gold_v2 strict candidate accuracy | 88.3% | 未见人工语言标签结果 |
| gold_v2 current symptom accuracy | 100.0% | 当前性规则结果 |
| gold_v2 preliminary intent accuracy | 53.3% | 仍是主要短板 |
| 真实模型 Schema valid rate | 96.30% | 合成工程评测 |
| 真实模型 slot precision / recall | 77.27% / 58.93% | 模型仍会错提与漏提 |
| Evidence grounding | 100% | 合成工程评测 |
| Risk preemption | 100% | 合成工程评测 |
| 诊断 / 用药泄漏 | 0 / 0 | 合成工程评测 |

上述均不是临床准确率。

### 9.2 分主诉发布原则

- 规则通过不等于真实 Provider 通过；
- 每个新增主诉必须拥有独立的槽位边界、风险规则、合成回归和人工验证证据；
- 未达到门槛时保持 `disabled` 或 `experimental`；
- 不使用参与调参的数据冒充独立验证结果。

## 10. 典型 Bad Case

| Bad Case | 失败模式 | 通用处理 | 当前状态 |
|---|---|---|---|
| “胸口发烧” | 局部发热感被误当体温发热 | 局部语境与歧义判断 | 不自动写入 fever |
| “昨天 38.5 度，现在已退烧” | 历史读数被写成当前体温 | `resolved` 状态与当前/最高温度分离 | 摘要显示本次已缓解 |
| “没有胸痛，但后来突然胸痛” | 全句否定导致风险漏判 | 定位每次 evidence 出现与转折后的当前语境 | 后半句触发风险 |
| “不是干咳，是有痰” | 极性和转折导致枚举冲突 | 候选词级上下文 | 识别 productive |
| `title/ask` 年龄或主题冲突 | 拼接或错配数据污染候选 | mismatch 门禁 | 不自动进入候选 |
| gold_v2 意图指标下降 | 开发集规则泛化不足 | 冻结规则并披露未见集下降 | 待改进 |
| 头痛/头晕槽位混淆 | 位置、体验、规律、诱因边界不稳定 | 分主诉 Provider 门禁 | 真实 Provider 禁用 |
| DeepSeek 输出截断 | 输出预算不足导致非法 JSON | 提高输出预算，Schema 不放宽 | 已修复并回归 |
| Strict Function Calling 返回 400 | Provider 实际能力不支持 | 缓存 unsupported，回退 JSON Object | 默认关闭 |
| 腹痛数据门禁未完全通过 | 边界证据和人工一致性不足 | 保守规则、固定澄清、实验性标识 | Provider 禁用 |

## 11. 验收标准

一次版本发布必须同时满足：

1. 当前范围内核心规则和页面自动化测试全部通过；
2. 风险表达在 Provider 前终止，调用次数为 0；
3. 否定、历史、已缓解和假设不被误写为当前；
4. 摘要不编造用户未提供的信息；
5. 诊断、药物、剂量和治疗泄漏为 0；
6. Provider 不可用时规则流程可独立完成；
7. 静态构建不包含 API Key、Provider Base URL 或 localhost API 地址；
8. 新增主诉只按独立门禁放行，不因总体指标被平均稀释。

## 12. 依赖、风险与开放问题

### 12.1 外部依赖

- 医疗专家对风险语言、产品边界和患者教育内容的审查；
- Provider 数据留存、隐私条款和区域合规确认；
- 生产环境共享限流、预算、审计和密钥轮换能力。

### 12.2 主要风险

- 规则词表无法覆盖全部自然语言；
- 用户可能误解为诊断或临床分诊；
- 意图路由在未见数据上泛化不足；
- 真实模型可能返回合法但语义错误的结构化结果；
- 小样本探索性测试不能证明普遍可用性。

### 12.3 开放问题

1. 摘要是否真的改善用户与人工服务或医生的沟通效率？
2. 7 轮上限在不同主诉下是否仍然合理？
3. 哪些风险表达必须由医疗专家重新定义？
4. 是否需要在取得正式内容责任机制后增加受控患者教育 RAG？
5. 真实 Provider 的收益是否足以抵消延迟、隐私和运营成本？
