# MedAsk 3—5 分钟面试演示话术

## 0:00—0:30｜背景

“很多患者并不是没有信息，而是不知道怎样把起病时间、当前状态、伴随症状和已经采取的措施组织起来。MedAsk 的定位是就医前信息整理，不做 AI 诊断，也不给药物或治疗建议。”

## 0:30—1:00｜数据问题

“我先做了独立数据治理。原始对话里有模板、重复、`title/ask` 错配，医生答案还可能包含不适合复用的诊疗内容。因此产品只使用患者侧文本做语言标签研究，并通过 120 条开发集和 60 条未见人工验证集建立 Human-in-the-loop 闭环。”

## 1:00—2:00｜普通问诊

1. 打开首页，指出范围和边界。
2. 在开发环境选择合成案例“普通发热”，说明一键按钮只填充、不自动提交。
3. 点击“开始整理”，展示主诉标签、轮次和标准问题。
4. 用自然语言回答；指出模型只提出槽位候选，状态机仍由 Harness 控制。
5. 若关闭服务端，展示系统无感回退标准输入。

## 2:00—2:30｜风险中断

选择合成案例“风险中断”：“我现在咳嗽，而且喘不上气”。提交后系统在调用模型前由风险规则中断普通流程。强调页面只给固定安全提示，不推断疾病。

## 2:30—3:00｜结构化摘要

完成普通案例，展示患者类型、主诉、起病时间、当前症状、伴随症状、已采取措施、尚未获取和用户暂不清楚。演示复制摘要和浏览器打印；指出摘要不补写用户没有提供的信息。

## 3:00—4:00｜架构与评测

“浏览器先做风险拦截，服务端重新计算槽位白名单，模型返回的 JSON 还要经过服务端 Schema、evidence 落地和前端 Harness 二次校验。真实 DeepSeek 三轮合成评测的 Schema 有效率是 96.30%，evidence grounding 和 risk preemption 都是 100%，诊断与药物泄漏为 0；但 slot recall 只有 58.93%，所以失败时必须回退规则，安全优先于填满信息。”

## 4:00—4:30｜产品反思

“这个项目最重要的不是接了模型，而是知道模型应该停在哪里。未见验证显示规则的主诉与当前性泛化较好，但意图路由退化明显；AI 也不能代替人工金标。下一步应先做隐私合规和真实用户可用性验证，而不是扩大疾病范围。”

> 演示全程只使用仓库内三个合成案例，不展示真实患者文本或人工金标行级数据。
