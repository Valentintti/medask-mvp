# MedAsk 未来改进方案

> 规划周期：未来 3—6 个月
> 规划基线：当前 GitHub Pages 规则版、受控 LLM Adapter 与实验性腹痛模块
> 说明：本文是产品与工程路线图，不是医疗能力扩张承诺。

## 1. 改进目标

下一阶段不以“增加更多主诉”或“让模型回答更多问题”为首要目标，而是补齐以下证据链：

1. 产品是否真的帮助用户完成就医前信息整理；
2. 风险语言和边界是否经过专业审查；
3. 意图路由能否在未见数据上稳定泛化；
4. 真实 Provider 是否在明确收益下保持可控；
5. 隐私、供应商留存和运营基础是否足以支持真实用户。

## 2. 当前基线与主要缺口

### 2.1 已完成能力

- 规则版支持发热、咳嗽、头痛、头晕，腹痛为实验性；
- 最多 7 轮，支持条件槽位、修改、返回、跳过和结构化摘要；
- 风险规则优先于 Provider；
- 模型只提出候选，Harness 控制状态与写入；
- 建立开发集、未见人工验证集、合成评测和 Bad Case 回归；
- 静态 GitHub Pages 无需 API 即可完成核心流程。

### 2.2 关键缺口

| 缺口 | 现有证据 | 影响 |
|---|---|---|
| 意图路由泛化不足 | gold_v2 意图准确率 53.3%，`symptom_intake` precision 46.3% | 可能把随访、问药或知识咨询送入普通症状流程 |
| 真人研究样本小 | 探索性测试及复测均为 n=2 | 不能证明普遍可用性或摘要价值 |
| 医疗专业审查缺失 | 当前风险规则仅为保守 Demo 语言规则 | 不能声称临床安全或完整分诊 |
| 真实模型召回偏低 | 合成评测 recall 58.93% | 自然语言输入仍可能漏提，增加重复追问 |
| 分主诉 Provider 能力不一致 | 头痛、头晕和腹痛未通过真实 Provider 门禁 | 不能统一开放自然语言自动写入 |
| 生产运营能力有限 | 单进程限流、预算与审计 | 多实例、长期留存和成本管理不足 |

## 3. 改进原则

1. **安全指标优先于填充率**：宁可回退标准问题，也不接受无证据候选。
2. **规则通过不等于模型通过**：每个主诉独立评测和发布。
3. **开发集与验证集隔离**：冻结 gold，不针对未见集继续调参。
4. **先验证用户价值，再扩大能力**：先证明摘要有用，再增加主诉。
5. **AI 不生成金标**：AI 只做预标注，人工负责最终判断。
6. **失败必须可见**：保留 Bad Case、禁用状态和停止条件。
7. **不为技术展示强行加入 RAG 或 Function Calling**：只有明确产品收益和稳定 Provider 能力时启用。

## 4. 优先级路线图

### P0：专业边界、隐私与责任基础

目标：让产品具备进入更大规模测试前的最低责任基础。

#### 工作项

1. 邀请医疗专业人员审查：
   - 风险提示文案；
   - 当前主诉的风险边界；
   - “不支持范围”与升级提示；
   - 结构化摘要是否可能造成误导。
2. 完成隐私告知与同意流程设计：
   - 是否发送健康文本到第三方 Provider；
   - 数据最小化、保留周期和删除机制；
   - 用户关闭自然语言辅助的选择。
3. 审查 Provider 条款：数据留存、训练使用、区域、日志和故障处理。
4. 定义严重事件与回滚流程：风险漏判、越界输出、密钥泄露和供应商异常。

#### 放行门槛

- 所有风险文案和边界完成专业审查并版本化；
- 真实 Provider 使用前获得明确知情同意；
- 诊断、药物和治疗泄漏回归保持为 0；
- 风险命中始终在网络请求前终止。

### P1：用户价值与可用性验证

目标：证明摘要和 7 轮流程对目标用户有实际帮助。

#### 工作项

1. 进行 8—12 名目标用户的形成性可用性测试；这是建议样本，不是统计代表性承诺。
2. 覆盖不同数字素养、手机操作习惯和表达风格。
3. 重点观察：
   - 是否理解“信息整理而非诊断”；
   - 是否能在 7 轮内完成；
   - 是否理解否定、历史、已缓解和跳过信息；
   - 摘要是否帮助用户发现遗漏；
   - 用户是否愿意复制或展示摘要。
4. 与人工接诊或模拟接诊人员开展摘要可交接性评审，不收集真实医疗结论。

#### 建议指标

- 核心流程任务完成率；
- 风险和不支持页面到达率；
- 修改、返回和清空操作发现率；
- 摘要事实忠实率；
- 需要人工解释的字段和页面；
- 用户是否准确复述产品边界。

不使用满意度分数替代事实性任务指标，也不把小样本结果表述为普遍可用性。

### P1：意图路由重构

目标：减少症状预问诊与随访、问药、报告解读、医院费用和知识咨询之间的混淆。

#### 工作项

1. 重新定义互斥主意图与可并存的 secondary flags。
2. 针对错误类型定向补充样本，而不是重新随机审核大批数据。
3. 单独建立以下边界集：
   - 已确诊和治疗后咨询；
   - 吃什么药、能不能吃、剂量和药物作用；
   - 具体指标、化验单和影像解释；
   - 挂什么科、哪家医院和费用；
   - 疾病定义、病因和遗传等知识问题。
4. 冻结新版规则后创建新的未见验证集，禁止继续使用 gold_v2 调参。

#### 建议门槛

- `symptom_intake` precision 明显高于当前 46.3%；
- 不通过大规模拒绝样本制造虚假高精度；
- 各高特异意图有明确 precision、recall 和错误类型；
- `uncertain` 单独报告，不自动当作错误或正确。

### P1：真实 Provider 可靠性

目标：证明 LLM 的增量价值，而不是证明系统“接上了模型”。

#### 工作项

1. 保持真实 Provider 仅开放已经验证的发热与咳嗽。
2. 为每个主诉建立独立 extraction schema、枚举说明和合成边界集。
3. 评估模型是否减少重复追问、缩短填写步骤，而不仅是槽位分数。
4. 增加模型、Prompt、Schema 和 Git 版本记录，保证评测可复现。
5. 持续验证超时、取消、迟到响应、冲突、低置信和非法输出降级。

#### 发布门槛

- Schema valid ≥95%；
- Evidence grounding ≥95%；
- Risk preemption =100%；
- Invalid-output rejection =100%；
- 诊断与用药泄漏 =0；
- 该主诉 precision、recall 和 exact match 达到预设门槛；
- 与纯规则相比存在可解释的用户体验收益。

### P2：受控 RAG 探索

RAG 不进入当前诊断或风险链路。只有在确认用户需要“就诊准备说明”或“审核过的患者教育信息”后，才开展小范围探索。

#### 前置条件

- 建立由专业人员审核的知识库；
- 每条内容具有来源、适用范围、版本、更新时间和过期状态；
- 明确内容负责人和撤回机制；
- 设计“无结果”与“来源冲突”降级；
- 完成引用准确率、过期内容和错误检索评测。

#### 允许的输出

- 就诊前需要准备的信息；
- 如何阅读结构化摘要；
- 产品支持范围和操作说明；
- 审核过的通用患者教育材料。

#### 始终禁止

- 用检索结果生成诊断、疾病概率、处方或治疗方案；
- 让 RAG 覆盖风险规则；
- 没有来源时让模型补写答案；
- 把检索命中率表述为临床正确率。

### P2：Function Calling 能力治理

当前 Strict Function Calling 已适配但默认关闭。后续只有在目标 Provider 和模型版本稳定支持时才重新启用。

#### 工作项

- 启动时进行非敏感能力探测；
- 明确不支持后在当前进程缓存 `unsupported`；
- 工具调用缺失或非法时最多回退一次 JSON Object；
- Function Calling 和 JSON Object 共用同一套 Schema、evidence 与 Harness 校验；
- 不让工具调用触发诊断、药物或外部医疗系统动作。

Function Calling 的成功标准是减少结构失败与延迟，而不是增加模型权限。

### P2：生产运营能力

1. 将单进程限流和预算迁移到共享存储或 API Gateway。
2. 建立请求成本、失败类别和降级率看板，只记录非敏感元数据。
3. 增加密钥轮换、供应商切换和故障演练。
4. 定义日志保留、审计访问和删除策略。
5. 建立版本发布、功能开关、分主诉回滚和灰度机制。

## 5. Bad Case 驱动的改进闭环

### 5.1 分类

| 类型 | 示例 | 处理方向 |
|---|---|---|
| 数据错误 | 模板、拼接、`title/ask` 冲突 | 数据门禁、来源标记和人工抽检 |
| 语境错误 | 否定、历史、已缓解、假设和转折 | 候选词级局部上下文 |
| 槽位边界错误 | 头痛位置、体验、规律混淆 | 独立槽位定义和分主诉评测 |
| 意图错误 | 随访、问药或知识咨询进入症状流程 | 高特异意图优先与新未见集 |
| 结构错误 | 截断 JSON、缺字段、额外字段 | 严格 Schema、输出预算和安全拒绝 |
| Provider 能力错误 | Strict Function Calling 不支持 | 能力缓存、一次回退和功能开关 |
| 发布证据不足 | 腹痛门禁未完全通过 | `experimental`、固定澄清和 Provider 禁用 |

### 5.2 标准流程

```text
发现失败
  ↓ 脱敏归类，不保存患者原文
判断是数据、规则、模型、Provider 还是产品问题
  ↓
只修复通用根因，不为单个 ID 写特例
  ↓
确定性测试 → 小型门禁 → 未见验证
  ↓
通过：分主诉放行
未通过：禁用 / experimental / 回退规则
```

### 5.3 停止条件

- 固定开发集继续优化但未见集不提升；
- 某指标提升同时明显伤害风险、安全或其他主诉；
- 需要降低 Schema、evidence 或 Acceptance Policy 才能通过；
- 继续调参只能为特定样例增加例外；
- 尚无真实用户价值证据却持续扩大医疗范围。

## 6. 90 天建议计划

| 阶段 | 时间 | 核心交付 | 退出条件 |
|---|---|---|---|
| 阶段 1 | 第 1—3 周 | 专业审查清单、隐私与同意方案、风险规则版本 | P0 问题有负责人和关闭标准 |
| 阶段 2 | 第 4—6 周 | 形成性可用性研究、摘要交接评审、问题优先级 | 关键流程问题完成一轮修复与复测 |
| 阶段 3 | 第 7—9 周 | 意图标签 V3、定向边界集、新未见集方案 | 不再使用现有 gold 调参 |
| 阶段 4 | 第 10—12 周 | 发热/咳嗽 Provider 增量价值评测、发布建议 | 安全门槛通过且证明相对规则的实际收益 |

RAG、更多主诉和真实人群开放不属于本 90 天默认承诺，必须通过独立立项与门禁。

## 7. 版本决策建议

### 保持上线

- GitHub Pages 纯前端规则 Demo；
- 发热、咳嗽、头痛和头晕规则版；
- 风险前置、7 轮流程、编辑与结构化摘要。

### 保持实验性

- 腹痛规则版和 Mock；
- 所有尚未完成专业审查的风险语言。

### 保持禁用

- 头痛、头晕和腹痛真实 Provider 自动接受；
- 胸部不适普通流程；
- 医学知识 RAG；
- 任何诊断、药物、剂量、检查或治疗输出。

## 8. 成功定义

下一阶段成功不等于“模型分数更高”或“支持更多症状”，而是：

1. 用户能准确理解产品边界并独立完成流程；
2. 摘要对信息交接有可观察价值；
3. 风险和越界输出继续由确定性规则控制；
4. 意图路由在新的未见集上改善；
5. 模型相对纯规则提供可量化的体验收益；
6. 证据不足的能力仍能被诚实地禁用或标记实验性。
