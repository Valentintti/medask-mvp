# MedAsk 产品案例：从医疗对话数据治理到受控预问诊 Demo

## 1. 背景与用户痛点

患者在就医前常常知道“哪里不舒服”，却难以按时间、程度、伴随症状和已采取措施组织信息。MedAsk 将这一问题定义为**信息整理**，帮助用户形成可交接的结构化摘要，而不是尝试给出疾病结论。

## 2. 为什么不做 AI 诊断

诊断、用药和治疗建议需要更高等级的临床证据、合规体系与专业责任。本项目把范围收窄到 18—65 岁成人的发热和咳嗽，固定风险表达由确定性规则前置拦截，模型只能提出已有槽位的候选值。这样可以验证交互与工程闭环，同时避免把语言模型能力包装成临床能力。

## 3. 数据集暴露出的现实问题

数据工程阶段发现了四类关键问题：大量占位符和模板文本；`title` 与 `ask` 年龄、主题或语义不一致；相同文本和模板变体重复；历史医生 `answer` 可能包含诊断、用药或治疗内容，不适合作为患者端产品回答依据。因此产品规则只迁移抽象后的标签定义和方法，不复制原始患者文本、金标 CSV 或医生答案。

## 4. 为什么只使用患者侧文本

候选检索、人工审核和产品输入只基于患者侧 `title/ask`。医生答案不进入患者端生成链路，避免把历史回答当作可靠诊疗建议，也让评测目标保持在“文本是否有效、症状是否当前、主诉和咨询意图”四个语言标签上。

## 5. Human-in-the-loop 标注闭环

流程依次完成：只读盘点 → 内科数据画像 → 透明规则候选 → 120 条人工开发集 `gold_v1` → V2 规则冻结 → 60 条未见人工验证集 `gold_v2` → AI 双轮预标注与人工盲审。AI 标签始终保存在独立字段，冲突、低置信、否定、历史、儿童孕产妇和模板错配进入人工队列；AI 一致不等于正确。

## 6. gold_v1 与 gold_v2

- `gold_v1` 是参与规则开发的 120 条平衡开发集。V2 的严格候选准确率为 86.7%，当前性规则准确率为 98.2%，意图准确率为 67.5%。
- `gold_v2` 是未参与 V2 调参的 60 条人工未见验证集。严格候选准确率为 88.3%，当前性规则准确率为 100.0%；但意图准确率降至 53.3%，`symptom_intake` precision 为 46.3%。

结论是主诉与当前性规则具备一定泛化能力，意图路由仍有明显分布偏移和局部过拟合信号。这些都是语言标签结果，不是临床准确率。

## 7. 规则版与 LLM 版架构

规则版由风险引擎、主诉识别、槽位引擎、摘要引擎和状态机组成。LLM 版没有替换它，而是在相同 Harness 外围增加候选提取器与非风险问题改写器。服务端重新计算槽位白名单，严格验证 JSON；前端继续做 Schema、evidence 局部上下文和槽位约束校验。

```text
用户输入 → 前端风险规则 → 服务端安全代理 → 模型候选
       ↘ 命中即升级       ↓ Schema / evidence 校验
                    Harness 二次校验 → 会话状态 / 摘要
```

## 8. Harness 为什么控制状态机

模型不能修改 `status`、`turnCount`、`maxTurns`、`required` 或 `showWhen`，也不能写入风险槽位。只有当前肯定、证据在原文、置信度达到门槛且通过类型校验的候选才可能被 Harness 接受；已有值冲突转为固定澄清问题。

## 9. 模型失败如何降级

风险规则始终先于网络调用。模型不可用、超时、限流、预算耗尽、Schema 非法、evidence 不落地或输出越界时，系统显示统一提示并继续标准问题。DeepSeek Strict Tool 默认关闭，当前使用 `json_object` 后仍经过全部安全校验；显式开启后若首次明确不支持，本进程不再重复尝试。

## 10. 当前范围与限制

当前只支持 18—65 岁成人、发热和咳嗽；最多 7 轮；风险规则是保守 Demo 条件，不是完整临床分诊标准。真实模型合成评测衡量的是结构化语言工程质量，不能证明诊断、安全或医疗有效性。

## 11. 后续路线图

1. 先完成隐私告知、同意、供应商留存与合规审查。
2. 用合成和定向人工案例持续做安全回归，不扩大诊断能力。
3. 做可用性与无障碍测试，验证摘要是否真正帮助沟通。
4. 生产环境升级到共享限流、预算、审计与密钥轮换基础设施。
5. 只有在新增主诉有独立数据证据和风险规则后，才逐类扩展范围。
